# Phase 與 Priority 正確匹配指南

**版本：** 1.0  
**最後更新：** 2025-12-22  
**目的：** 確保 AI 在從 plan 分解任務時，能正確匹配階段（Phase）與優先級（Priority）

---

## 🚨 核心問題

在任務分解過程中，**Phase（階段）** 與 **Priority（優先級）** 必須正確匹配。

**❌ 常見錯誤：**

- Phase 4（最後階段）包含緊急任務
- 高優先級任務被分配到後期 Phase
- Phase 編號與實際執行順序不符

**✅ 正確做法：**

- Phase 編號應該反映任務的優先級和依賴順序
- 緊急和高優先級任務應該在早期 Phase 完成
- 後期 Phase 應該只包含低優先級或建議執行的任務
- **(新)** 務必使用 Phase 任務命名格式：`...-phase-[N]-...` (詳見 [PLANS_RULES.md](./PLANS_RULES.md))

---

## 📐 Phase 與 Priority 匹配原則

### Phase 1（第一階段）：基礎建設與核心功能

**適合的優先級：**

- 🔴 緊急 (1-urgent)
- 🟠 高 (2-high) - 核心基礎任務

**任務特性：**

- 專案的核心基礎設施
- 其他任務的關鍵依賴
- 必須最優先完成的緊急任務
- 資料模型、核心介面定義

**範例任務：**

- 更新 TypeScript 資料模型介面
- 實作核心計算工具函式
- 建立基礎資料庫 schema
- 設定專案架構

---

### Phase 2（第二階段）：服務層與業務邏輯

**適合的優先級：**

- 🟠 高 (2-high)
- 🟡 中 (3-mid) - 核心業務邏輯

**任務特性：**

- 依賴 Phase 1 完成的基礎建設
- 核心業務邏輯實作
- API 服務層開發
- 資料處理與驗證

**範例任務：**

- 更新食材服務層
- 實作 API 端點
- 資料遷移腳本開發
- 業務規則實作

---

### Phase 3（第三階段）：UI 層與使用者介面

**適合的優先級：**

- 🟡 中 (3-mid)
- 🟢 低 (4-low)

**任務特性：**

- 依賴 Phase 1 和 Phase 2 的成果
- UI 元件開發
- 使用者介面更新
- 部分可並行開發

**範例任務：**

- 更新食材表單 Modal
- 更新食材列表元件
- 畫面調整與優化
- 互動功能實作

---

### Phase 4+（第四階段及以後）：整合、測試與優化

**適合的優先級：**

- 🟢 低 (4-low)
- 💡 建議執行 (5-suggest)

**任務特性：**

- 依賴所有前置任務完成
- 測試與驗證
- 效能優化
- 文件撰寫
- 額外功能建議

**範例任務：**

- 單元測試撰寫
- 效能優化
- 文件更新
- 額外功能開發

---

## ⚠️ 特殊情況處理

### 情況 1：最終階段的關鍵任務

某些任務因為**依賴關係**必須放在最後階段執行，但**優先級卻很高**（甚至是緊急）。

**典型範例：**

- 整合測試與驗證（必須等所有開發完成）
- 正式資料遷移（必須等測試通過）
- 生產環境部署（必須等所有驗證完成）

**正確處理方式：**

1. **在 TASK_OVERVIEW 中明確標註**

```markdown
### Phase 4: 整合與驗證

| 編號    | 優先級  | 類別 | 任務名稱       | 狀態    | 備註                        |
| ------- | ------- | ---- | -------------- | ------- | --------------------------- |
| `In7T8` | 🔴 緊急 | test | 整合測試與驗證 | 📋 待辦 | ⚠️ 最終階段關鍵任務         |
| `Pr8M9` | 🔴 緊急 | feat | 執行正式遷移   | 📋 待辦 | ⚠️ 需等測試通過，高風險操作 |
```

2. **在任務描述中說明原因**

```markdown
**任務名稱:** 整合測試與驗證

**優先級:** 緊急

**說明:**
雖然此任務優先級為緊急，但因為需要等待所有前置任務完成才能執行，
因此被分配在 Phase 4（最終階段）。

此任務是專案成功的關鍵，必須確保：

1. 所有前置任務盡快完成
2. 預留充足時間進行測試
3. 發現問題時能立即修正
```

3. **確保前置任務優先級正確**

如果最終階段有緊急任務，所有前置任務的優先級都應該相應提高，確保能盡快完成。

## ✅ 驗證檢查清單

完成任務分解後，請進行以下檢查：

### 1. Phase 優先級分布檢查

- [ ] **Phase 1** 是否包含所有緊急任務？（特殊情況除外）
- [ ] **Phase 1** 是否包含核心高優先級任務？
- [ ] **Phase 2-3** 的優先級是否逐步降低？
- [ ] **Phase 4+** 是否只包含低優先級或建議執行任務？（特殊情況除外）

### 2. 特殊情況檢查

- [ ] 如果後期 Phase 出現高/緊急任務，是否有明確標註？
- [ ] 是否在任務描述中說明了原因？
- [ ] 前置任務的優先級是否已相應提高？

### 3. 依賴關係檢查

- [ ] Phase 之間的依賴關係是否合理？
- [ ] 緊急任務的前置任務是否都已標記為高優先級？
- [ ] 是否有循環依賴？

### 4. 並行性檢查

- [ ] 每個 Phase 內是否有可並行執行的任務？
- [ ] 並行任務之間是否沒有依賴關係？
- [ ] 是否充分利用了並行執行的可能性？

## 📊 正確範例 vs 錯誤範例

### ❌ 錯誤範例

```markdown
### Phase 1: 基礎建設

| 編號 | 優先級 | 任務名稱     |
| ---- | ------ | ------------ |
| Ts01 | 🟡 中  | 更新資料模型 |
| Ex02 | 🟢 低  | 實作計算工具 |

### Phase 4: 整合與驗證

| 編號 | 優先級  | 任務名稱     |
| ---- | ------- | ------------ | --------------------------------- |
| In01 | 🔴 緊急 | 整合測試     |
| Pr02 | 🔴 緊急 | 正式遷移     |
| Ui03 | 🔴 緊急 | 更新 UI 元件 | ← **錯誤！UI 更新不應在 Phase 4** |
| Sv04 | 🟠 高   | 更新服務層   | ← **錯誤！服務層不應在 Phase 4**  |
```

**問題：**

1. Phase 1 的任務優先級太低（應該是緊急或高）
2. Phase 4 包含太多緊急任務
3. UI 和服務層任務不應該在最後階段

---

### ✅ 正確範例

```markdown
### Phase 1: 基礎建設（可並行）

| 編號    | 優先級 | 任務名稱     | 備註            |
| ------- | ------ | ------------ | --------------- |
| `Ts01x` | 🟠 高  | 更新資料模型 | 最優先執行      |
| `Ex02y` | 🟠 高  | 實作計算工具 | 可與 Ts01x 並行 |

### Phase 2: 服務層

| 編號    | 優先級 | 任務名稱     | 依賴         |
| ------- | ------ | ------------ | ------------ |
| `Sv03z` | 🟠 高  | 更新服務層   | Ts01x, Ex02y |
| `Mg04w` | 🟡 中  | 實作遷移腳本 | Ts01x, Ex02y |

### Phase 3: UI 層（可部分並行）

| 編號    | 優先級 | 任務名稱     | 依賴         |
| ------- | ------ | ------------ | ------------ |
| `Ui05v` | 🟡 中  | 更新食材表單 | Ts01x, Ex02y |
| `Ui06u` | 🟡 中  | 更新食材列表 | Ts01x, Ex02y |

### Phase 4: 整合與驗證

| 編號    | 優先級  | 任務名稱     | 備註                  |
| ------- | ------- | ------------ | --------------------- |
| `Ts07t` | 🟠 高   | Dry-Run 測試 | 需 Mg04w              |
| `In08s` | 🔴 緊急 | 整合測試     | ⚠️ 最終階段關鍵任務   |
| `Pr09r` | 🔴 緊急 | 正式遷移     | ⚠️ 需測試通過，謹慎！ |
```

**優點：**

1. Phase 1 包含核心高優先級任務
2. Phase 逐步推進，優先級合理降低
3. Phase 4 的緊急任務有明確標註和說明
4. 依賴關係清晰，可並行性良好

---

## 🎯 總結

**關鍵原則：**

1. **Phase 編號 ≈ 執行順序 ≈ 優先級降低**
2. **緊急任務應在早期 Phase，除非有不可避免的依賴**
3. **特殊情況必須明確標註和說明**
4. **完成分解後必須進行驗證檢查**

**記住：**

- Phase 1-2：🔴 緊急 + 🟠 高
- Phase 3：🟡 中 + 🟢 低
- Phase 4+：🟢 低 + 💡 建議（+ 特殊緊急任務需標註）

遵循這些原則，可以確保任務分解合理、執行順序清晰、優先級正確！
