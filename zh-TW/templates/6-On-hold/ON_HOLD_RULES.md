## 🎯 資料夾用途

此資料夾存放**暫時無法繼續處理**的任務。任務可能因為需求不清楚、等待其他系統重構、需要重新設計、或其他阻礙因素而暫停。

### 📁 資料夾組織規則

⭐ **重要:** 為避免不同 plan 的任務混在一起，任務必須按照所屬 plan 組織：

1. **任務必須放在日期-plan 編號-feature 名稱資料夾中**
   - 例如: `6-On-hold/2025-12-01-[plan-Xm1q2]-payment-integration/2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-stripe-integration.md`
2. **從其他階段移動任務時，保持相同的日期和 plan 資料夾結構**
3. **⚠️ 資料夾命名格式:** `[YYYY-MM-DD]-[plan-xxxxx]-[feature-name]`
   - ❌ 錯誤: `6-On-hold/2025-12-01-[plan-Xm1q2]/`（缺少 feature-name）
   - ✅ 正確: `6-On-hold/2025-12-01-[plan-Xm1q2]-payment-integration/`

**範例結構:**

```
6-On-hold/
├── 2025-12-01-[plan-Xm1q2]-payment-integration/
│   └── 2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-stripe-integration.md
├── 2025-12-05-[plan-Rt7w3]-user-authentication/
│   └── 2025-12-05-[plan-Rt7w3]-1-urgent-[task-Bc8K7]-feature-jwt-implementation.md
├── 2025-12-31-[no-spec]-bugfix-login/
│   └── 2025-12-31-[no-spec]-1-urgent-[task-Bz8k3]-bug-fix-login-timeout.md
└── .on-hold-task-template.md
```

### 📁 無對應 Spec 的任務

對於使用 `[no-spec]` 命名格式的任務：

- **無論是單一任務文件或多個任務文件，都必須放在共同父層資料夾中**
- **可以直接在 6-On-hold/ 建立**
- **跨階段移動時必須保留父層資料夾結構**

**詳細規範：** 請參考 [COMMON_CONVENTIONS.md](../COMMON_CONVENTIONS.md#無對應-spec-的-plan-文件)

## 📋 何時將任務移到 6-On-hold

任務在以下情況下應移到 `6-On-hold/`：

### 情況 1: 需求不清楚或描述不完整

- 開發過程中發現任務描述不夠詳細
- 需要與相關人員澄清需求
- 需求本身存在矛盾或不合理之處

### 情況 2: 等待相依任務完成

- 發現必須先完成其他任務
- 等待其他系統重構或 API 更新
- 等待第三方服務或套件更新

**⚠️ 重要：在暫停前先考慮重新拆分**

在將任務因相依性問題移到 6-On-hold 之前，**優先考慮**以下解決方案：

1. **拆分成獨立部分**
   - 評估任務是否可以拆分成「不依賴其他任務的部分」和「依賴其他任務的部分」
   - 將獨立部分拆分成新任務，可以立即開始處理
   - 只將真正依賴的部分保留在 6-On-hold

2. **使用介面/Mock 解除阻塞**
   - 如果等待的是其他任務的 API 或函數，先定義介面和 Mock 實作
   - 使用 Mock 版本繼續開發，等真實實作完成後再替換
   - 將「整合真實實作」拆分為獨立的後續任務

3. **調整任務順序**
   - 與用戶討論是否可以調整任務優先級
   - 將被依賴的任務提升為高優先級，優先完成
   - 避免多個任務因同一個依賴而全部暫停

**範例：**

```
原任務: `2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-user-dashboard.md` (需要等待 user-api 完成)

↓ 拆分為 ↓

新任務 1: `2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-user-dashboard-ui.md` (可立即開始，使用 Mock 資料)
新任務 2: `2025-12-01-[plan-Xm1q2]-3-mid-[task-B2c3D]-feature-user-dashboard-integration.md` (等待 user-api 完成後整合)

結果：1 個任務可以立即並行處理，只有 1 個任務需要等待
```

### 情況 3: 技術方案需要重新評估

- 當前技術方案遇到無法解決的問題
- 需要研究替代方案
- 需要進行技術驗證 (PoC)

### 情況 4: 多次測試失敗

- 在 `5-Re-testing/` 中修正超過 5 次仍失敗
- 發現根本問題需要重新設計

### 情況 5: 資源或優先級調整

- AI 工具額度耗盡，需等待恢復
- 優先級降低，暫時擱置
- 需要等待特定 AI 工具（如 Claude）額度恢復
- 需要等待特定 AI 工具（如 Claude）額度恢復

**任務完成時必須使用 `[✓]` 標記,不要使用 `[x]`**

- ✅ 正確: `- [✓] 已完成的項目`
- ❌ 錯誤: `- [x] 已完成的項目`

## 🔄 On-Hold 任務的管理流程

### 定期檢視 (建議每週)

1. **檢視所有 on-hold 任務**
2. **確認恢復條件是否滿足**
3. **更新任務狀態**
4. **決定下一步行動**

### 恢復任務的步驟

當恢復條件滿足時：

1. **更新任務檔案**
   - 記錄「恢復時間」
   - 更新「任務描述」（如有需求變更）
   - 重新評估「優先級」和「複雜度」

2. **決定移動目標**
   - 如果只是暫停，程式碼未開始 → 移回 `2-Plans/`
   - 如果程式碼已部分完成 → 移回 `3-Progressing/`
   - 如果程式碼完成但測試未過 → 移回 `5-Re-testing/`

3. **更新相依任務**
   - 檢查是否有其他任務因此任務暫停而暫停
   - 一併恢復相依任務

## ⚠️ 注意事項

### On-Hold 不是垃圾桶

- On-Hold 的任務仍然是有效的任務
- 定期檢視確保不會被遺忘
- 如果任務確定不做了，應該移到 `cancelled/`（可考慮新增此資料夾）

### 長時間 On-Hold 的任務

- **超過 1 個月:** 重新評估任務的必要性
- **超過 3 個月:** 考慮取消或重新規劃

### 相依任務的處理

- 當父任務 on-hold 時，子任務也必須 on-hold
- 在父任務恢復前，不應該處理子任務
- 明確記錄相依關係，避免遺漏

## 📊 On-Hold 統計與分析

定期檢視 `6-On-hold/` 資料夾，可以：

1. **識別常見阻礙** - 哪些類型的問題最常導致任務暫停
2. **改進需求流程** - 如果很多任務因需求不清而暫停，應改進需求澄清流程
3. **管理相依性** - 識別哪些系統或模組經常阻礙其他任務
4. **優化資源分配** - 了解 AI 工具額度的使用模式

## ✅ 恢復任務時的動作

當任務準備恢復時：

1. **更新任務檔案**

   ```markdown
   **恢復時間:** [填入實際當前時間，格式: YYYY-MM-DD HH:mm]

   **恢復說明:**
   [說明恢復條件如何滿足，以及任何需求或計畫的變更]
   ```

2. **重新評估任務**
   - 優先級是否仍然相同？
   - 複雜度評估是否需要調整？

3. **🚪 Stage Exit Checkpoint & 檔案移動**
   - **✅ Stage Exit Checkpoint（強制）**：離開 `6-On-hold` 前必須確認恢復說明與狀態已更新。
   - 將檔案 **移動（mv）** 到對應的資料夾（`2-Plans/`、`3-Progressing/` 或 `5-Re-testing/`）。
   - **🚪 Stage Entry Gate（強制）**：移動後確認 `6-On-hold/` 中該批次文件已不存在，無殘留檔案則刪除空資料夾。

4. **停下動作，等待確認**
