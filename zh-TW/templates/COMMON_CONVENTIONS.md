## ⚠️ 嚴格禁止規則：任務檔案移動與狀態變更

### 🚫 絕對禁止自動移動任務或變更狀態

**⚠️ 這是 Kanban 系統的最高優先級規則，違反此規則將導致工作流程中斷**

### 禁止動作清單

無論在任何階段（specs、plans、3-Progressing、testing、re-testing、done、on-hold），AI 工具**絕對不可擅自**執行以下動作：

❌ **嚴格禁止：**

1. **不得移動 Plan 檔案到其他階段資料夾**
   - 不得從 `2-Plans/` 移到 `3-Progressing/`
   - 不得從 `3-Progressing/` 移到 `4-Testing/`
   - 不得從 `4-Testing/` 移到 `7-Done/` 或 `5-Re-testing/`
   - 不得從任何階段移到 `6-On-hold/`
   - 不得從任何階段移到 `8-Archived/`

2. **不得修改 Plan 檔案中的「狀態」欄位**
   - 不得將狀態從「待辦 (To Do)」改為「處理中 (Progressing)」
   - 不得將狀態從「處理中 (Progressing)」改為「測試中 (Testing)」
   - 不得將狀態從「測試中 (Testing)」改為「完成 (Done)」
   - 不得將狀態改為「暫停 (On-Hold)」

3. **不得自行判斷任務已完成並移動**
   - 即使程式碼已完成
   - 即使測試已通過
   - 即使所有子步驟都完成

### ✅ 正確做法

**唯一例外：使用者明確指示**

只有當使用者**明確且清楚**地說出以下指令時才可執行：

✅ **允許的指令範例：**

- 「請將這個 Plan 移到 3-Progressing」
- 「幫我把 Plan 移到 4-Testing 階段」
- 「這個 Plan 可以標記為 done 了」
- 「請更改 Plan 狀態為 on-hold」
- 「移動到下一個階段」

❌ **不算明確指示的情況：**

- 「任務完成了」（沒有明確說要移動）
- 「測試通過了」（沒有明確說要改狀態）
- 「可以了」（太模糊）
- 使用者沒有提到移動或狀態變更

### 🤖 自動化模式例外條款（允許自動移動）

為了支援可替換的自動化流程（包含 skill 或其他自動化工具），新增以下例外條款：

在符合下列條件時，允許 AI 自動執行任務檔案移動與狀態變更：

1. 使用者明確啟用 **自動化模式**
2. 使用者明確授權 **自動轉階段**（例如：`auto_transition: true`）
3. 使用者明確選擇執行設定（由對應整合文件定義）

未同時滿足以上條件時，一律回到手動模式（manual mode），維持本文件原本的禁止規則。

#### 自動化整合文件（可替換）

`COMMON_CONVENTIONS.md` 僅定義長期不變的共通原則；任何工具特定設定（例如 `mode`、`pipeline`、`auto_transition` 組合與問答流程）統一放在整合文件，避免和特定 skill 綁死。

- 預設整合文件：`templates/SKILL_INTEGRATION.md`
- 若未來 skill 淘汰或替換，只需更新整合文件，不需要大幅改動本文件。

#### 失敗處理與回復流程（共通）

若流程失敗，依序執行：

1. 立即停止自動流程，保留現況，不做額外移動。
2. 回報失敗點（步驟、檔案、命令、錯誤摘要）。
3. 說明是否已寫入部分內容（避免使用者誤判為全失敗）。
4. 提供最小回復方案（例如：回到 `plan-only`、僅補測試、先修環境）。
5. 取得使用者確認後再繼續，不得自行重跑 full pipeline。

#### 自動化模式安全停止條件

若發生任一情況，必須立即停止自動流程並回報：

- 測試失敗
- 進入 `4-Testing` 後未建立測試程式碼
- 進入 `4-Testing` 後未完成「全部測試通過（all green）」驗證
- 缺少必要環境設定或權限
- 需求不明確或規則衝突
- 使用者中途取消或要求切回 manual mode

## 🚪 Stage Entry Gate（階段進入閘門，強制）

每次任務文件從一個階段進入下一個階段時，**必須依序完成以下動作，缺一不可**：

1. **移動（mv）**：將文件從來源階段資料夾移動到目標階段對應資料夾。**必須使用 `mv`，嚴禁使用 `cp`（複製）**。
2. **更新狀態**：移動完成後，**立即**更新文件中的「狀態」欄位為目標階段狀態（例如：「處理中 (Progressing)」→「測試中 (Testing)」）。
3. **更新欄位**：依目標階段 RULES 的「必須修改的欄位」補齊所有欄位（例如：開始時間、AI 工具、進度 checklist）。
4. **清除來源**：確認來源階段中該批次文件已不存在（已被 `mv` 移走）。若來源資料夾中已無同批次文件（含 `PLAN_OVERVIEW`），刪除該空資料夾。
5. **確認完成**：以上四步全部完成後，才可開始該階段的實作工作（寫程式碼、寫測試、建 summary 等）。

### Stage Entry Gate 禁止行為

- ❌ 不得先做事（寫程式碼、寫測試），再補移動或更新。
- ❌ 不得用「複製（cp）」取代「移動（mv）」，導致來源殘留。
- ❌ 不得在移動後跳過欄位更新就直接開始實作。
- ❌ 不得移動文件後不清除來源資料夾中的殘留。

### 來源清理驗證（強制）

每次跨階段移動完成後，必須立即驗證來源已清除：

```bash
# 驗證範例（替換為實際的 spec-id 或 plan-id）
find <project>/{來源階段}/ -name "*<spec-id>*" -o -name "*<plan-id>*"
```

若有殘留，立即清除後才可繼續。

## ✅ Stage Exit Checkpoint（階段離開檢查點，強制）

在離開每個階段（將文件移動到下一階段）前，必須完成以下動作：

1. **勾選所有 checkbox**：將該階段所有適用的進度項目標記為 `[✓]`（例如：`Lint 檢查通過`、`完成開發`）。
2. **補齊工作結果描述**：在文件中記錄本階段的實作結果、測試結果、或處理內容。
3. **更新 PLAN_OVERVIEW**：若同批次有 `0-PLAN_OVERVIEW`，同步更新統計、狀態與最後更新時間。
4. **確認以上三項全部完成後**，才可執行移動到下一階段。

### Stage Exit Checkpoint 禁止行為

- ❌ 不得在 checkbox 未勾完時移動文件到下一階段。
- ❌ 不得在未記錄工作結果時移動文件到下一階段。
- ❌ 不得在 PLAN_OVERVIEW 未更新時移動文件到下一階段。

## 🧪 Testing 品質閘門（強制）

只要任務文件被移動到 `4-Testing/`（包含任何 pipeline：spec/plan/testing、或 `*-then-go`、`*-to-archived`），必須強制完成以下事項：

1. **建立或補齊測試程式碼**
   - 不可只更新任務文件或只移動檔案。
2. **執行完整測試指令**
   - 必須執行專案主要測試命令（例如 `npm test`、`pnpm test`、`yarn test`、`uv run pytest`）。
   - 不可僅執行單一測試檔就視為完成整體 testing。
3. **確認全部測試通過（all green）**
   - 若專案規範包含 lint/type-check，也必須同步通過。
4. **記錄完整測試證據**
   - 必須將實際測試指令、完整輸出、統計數據寫入 testing 文件。

### 禁止行為（Testing 品質閘門）

- ❌ 只把文件移到 `4-Testing/` 就宣稱「testing 完成」。
- ❌ 在測試未全數通過時，移動到 `7-Done/` 或 `8-Archived/`。
- ❌ 省略測試指令與輸出的記錄。

## 📘 PLAN_OVERVIEW 同步規範（強制）

若同一批次存在 `0-PLAN_OVERVIEW`，任何 plan 文件跨階段移動時，都必須同步更新與管理 overview。

### 必須執行的同步動作

1. **每次 plan 文件移動後，立即更新 overview 內容**
   - 更新任務狀態統計與百分比。
   - 更新任務清單中的狀態與備註。
   - 更新 `最後更新` 時間。

2. **來源資料夾只剩 overview 時，overview 必須跟著移動**
   - 當來源資料夾中，除了 `0-PLAN_OVERVIEW` 外的同批次文件都已移到下一階段，`0-PLAN_OVERVIEW` 必須移到同一目標資料夾。
   - 移動後需立即更新 overview 內文與路徑描述。

3. **plan 文件必須做雙次更新**
   - 第一次：跨階段當下，更新「目前階段狀態」。
   - 第二次：該階段工作完成後，更新「完成內容/測試證據/結束時間」。

4. **overview 必須維持目錄索引能力**
   - 使用者只看 overview，就能快速掌握其餘 plan 文件目前處理狀態。
   - 禁止讓 overview 長期停在已無同批次文件的舊階段資料夾。

5. **同批次單步推進只允許單一目標階段**
   - 同一批次在一次階段推進中，不可同時拆分到多個目標階段資料夾。
   - 若發生拆分需求，必須先更新 overview，並由使用者明確確認後再分批處理。

### 理由說明

1. **人工確認** - 確保每個階段轉換都經過人工檢查和確認
2. **防止跳過** - 避免跳過重要的測試或驗證步驟
3. **流程控制** - 使用者保持對任務流程的完全控制權
4. **錯誤預防** - 防止 AI 誤判任務狀態導致流程混亂

### 如果完成工作該怎麼做？

以下規則預設適用於手動模式。若已啟用並授權自動化模式，則依整合文件定義執行。

當 AI 工具完成被指派的工作後：

1. ✅ **更新 Plan 檔案中的進度欄位**（如「開發進度」、「測試進度」）
2. ✅ **標記完成的子步驟為 `[✓]`**
3. ✅ **記錄完成時間和負責的 AI 工具**
4. ✅ **告知使用者工作已完成**
5. 🛑 **停止所有動作，等待使用者下一步指示**

### 🔀 任務同步實作判斷規則（Plan / Testing 共用）

當 AI 工具讀取完 Plan 或 Testing 文件，並準備開始實作時，必須先做以下判斷：

1. 優先檢查任務清單中的「是否可並行執行」與「任務間的依賴關係」欄位。
2. 若存在可同時執行、且彼此無依賴衝突的任務，應開啟同步實作模式並同步開始處理。
3. 若無法同步開始實作，必須遵循前後依賴順序，逐步執行任務內容（先完成前置任務，再執行後續任務）。

### 🛠️ Plan 實作階段強制執行規範

當 Plan 開始實作時，必須遵循以下規則：

#### A. 自動化模式（工具中立）

定義：已明確啟用自動化模式，且依整合文件的流程執行任務（可由 skill 或其他自動化工具實作）。

1. 開始處理任務時，先依「任務同步實作判斷規則」安排執行方式，再依照任務需求建立正確且完整的程式碼與測試程式碼；處理完成後，更新任務文件中「這次任務做了什麼」。
2. 程式碼中的註解一律使用英文。
3. 任務處理完成後，回頭檢查文件內容，確認所有任務已執行，且至少重複檢查三次。
4. 確認任務內容皆已實作完成後，再檢查文件是否已針對完成內容更新，並有詳細記錄實作結果。
5. 任務完成標記一律使用 `[✓]`，嚴禁使用 `[x]`。

**Definition of Done（Plan / 自動化模式）**

- 功能程式碼完成且可執行。
- 對應測試已建立且通過。
- 任務文件已更新本次實作內容與結果。
- 已完成至少三輪檢查並確認需求一致。

#### B. 手動模式（未啟用自動化）

定義：未啟用自動化模式，或以一般手動對話方式處理任務。

1. 開始處理任務時，先依「任務同步實作判斷規則」安排執行方式，再依照任務需求建立正確且完整的程式碼與測試程式碼；處理完成後，針對任務文件只更新「這次任務做了什麼」。
2. 程式碼中的註解一律使用英文。
3. 任務處理完成後，回頭檢查文件內容，確認所有任務已執行，且至少重複檢查三次。
4. 確認任務內容皆已實作完成後，再檢查文件是否已針對完成內容更新，並有詳細記錄實作結果。
5. 任務完成標記一律使用 `[✓]`，嚴禁使用 `[x]`。

**Definition of Done（Plan / 手動模式）**

- 功能程式碼完成且可執行。
- 對應測試已建立且通過。
- 任務文件已更新本次實作內容與結果。
- 已完成至少三輪檢查並確認需求一致。

### 🧪 Testing 實作階段強制執行規範

當 Testing 階段開始實作時，必須遵循以下規則：

#### A. 手動模式（未啟用自動化）

定義：未啟用自動化模式，或以一般手動對話方式處理測試任務。

2. 建立完整測試程式碼，並確保所有測試都已通過。
3. 必須執行完整測試範圍（不可只跑單一測試檔），並將測試指令與完整輸出寫入 testing 文件。
4. 程式碼中的註解一律使用英文。
5. 任務處理完成後，回頭檢查文件內容，確認所有任務已執行，且至少重複檢查三次。
6. 確認任務內容皆已實作完成後，再檢查文件是否已針對完成內容更新，並有詳細記錄實作結果。
7. 任務完成標記一律使用 `[✓]`，嚴禁使用 `[x]`。

**Definition of Done（Testing / 手動模式）**

- 測試程式碼完成，且所有測試通過。
- 測試指令、完整輸出與統計資訊已記錄在 testing 文件。

- 任務文件未被移動且狀態未被修改。

#### B. 自動化模式（工具中立）

定義：已明確啟用自動化模式，且依整合文件的流程執行測試任務（可由 skill 或其他自動化工具實作）。

1. 開始處理任務時，先依「任務同步實作判斷規則」安排執行方式，再依照任務需求建立正確且完整的測試程式碼；處理完成後，更新任務文件中「這次任務做了什麼」。
2. 建立完整測試程式碼，並確保所有測試都已通過。
3. 必須執行完整測試範圍（不可只跑單一測試檔），並將測試指令與完整輸出寫入 testing 文件。
4. 程式碼中的註解一律使用英文。
5. 任務處理完成後，回頭檢查文件內容，確認所有任務已執行，且至少重複檢查三次。
6. 確認任務內容皆已實作完成後，再檢查文件是否已針對完成內容更新，並有詳細記錄實作結果。
7. 任務完成標記一律使用 `[✓]`，嚴禁使用 `[x]`。

**Definition of Done（Testing / 自動化模式）**

- 測試程式碼完成，且所有測試通過。
- 測試指令、完整輸出與統計資訊已記錄在 testing 文件。
- 任務文件完整記錄測試結果與本次工作內容。
- 已完成至少三輪檢查並確認需求一致。

## 版號管理規範

### 版本欄位

本文件與相關整合文件至少應維持以下欄位：

- `Version`
- `Last Updated`
- `Effective Date`（或整合文件的 `effective_date`）

### 版本升級建議

- `+0.0.1`：文字修正、範例修正、無流程變更
- `+0.1.0`：流程規則新增或調整（不破壞既有流程）
- `+1.0.0`：破壞性變更（會改變既有執行行為）

## 計畫完成標記規範

### ⚠️ 重要：統一使用 `[✓]` 標記

**在所有規格文件 (Specs) 和實作計畫 (Plans) 中，項目完成時必須使用 `[✓]` 標記，不要使用 `[x]`**

### 標記使用規則

- ✅ **正確：** `- [✓] 已完成的項目`
- ❌ **錯誤：** `- [x] 已完成的項目`
- ✅ **未完成：** `- [ ] 待完成的項目`
- ✅ **保持不變：** 文字中的 ✅ 符號（如「✅ 正確做法」）保持使用 ✅

### 範例

```markdown
## 開發進度

- [✓] 程式碼框架建立
- [✓] 核心邏輯實現
- [ ] 錯誤處理
- [ ] 完成開發

## 測試進度

- [✓] 測試檔案建立
- [✓] 單元測試撰寫
- [✓] 測試執行
- [✓] 測試通過
```

## 檔案命名與編號規範

### 🔖 編號系統概述

**⭐ 重要：Specs 文件和 Plans 文件都需要加上編號前綴**

**Plans (原 Tasks) 編號格式：** `[spec-xxxxx]-[plan-yyyyy]-優先級-類別-描述.md`

- 包含來源 Spec 的編號（對應 Spec 資料夾內的編號）
- 加上 `plan-` 前綴和計畫自己的獨立編號
- 可以追蹤 Plan 來自哪個 Spec

**Specs (原 Plans) 編號格式：** `[spec-xxxxx]-檔案名稱.md`

- 同一個 Spec 資料夾內的所有文件使用相同編號
- 編號前加上 `spec-` 前綴

### 編號格式

```
[編號]-原檔名.md
```

#### 編號組成規則

- **位數：** 5 碼
- **字元集：** 英文大小寫（A-Z, a-z）+ 數字（0-9）
- **每一位：** 從 62 種字元中隨機選擇（26 大寫 + 26 小寫 + 10 數字）
- **總組合數：** 62^5 = 916,132,832 種
- **碰撞機率：** 10,000 個檔案的碰撞率僅 5.5%

### 完整命名格式

#### Plans (實作計畫) 文件

[YYYY-MM-DD]-[spec-xxxxx]-[數字]-[優先級]-[plan-yyyyy]-[類別]-簡短描述.md

```

**⭐ 分階段（Phase）Plans 命名規則：**

若 PLAN_OVERVIEW 或 Specs 中將實作計畫明確劃分為不同階段（Phase），**必須**在檔名中加入階段標記，以便按執行順序排序。

**Phase 格式:** `[YYYY-MM-DD]-[spec-xxxxx]-phase-[N]-[數字]-[優先級]-[plan-yyyyy]-[類別]-簡短描述.md`

- **規則：**
  - `phase-[N]`: N 為階段數字 (1, 2, 3...)
  - 若有區分 Phase，則**所有**相關 Plans 都應加上 `phase-[N]`
  - `phase-[N]` 置於 `[spec-xxxxx]` 之後、`[數字]` 之前
  - **範例：**
    - `2025-12-05-[spec-Ax4m2]-phase-1-1-urgent-[plan-Az4a2]-feature-auth.md`
    - `2025-12-05-[spec-Ax4m2]-phase-2-2-high-[plan-By3n1]-feature-profile.md`
- **排序優勢：** 檔案會先依照 Phase 排序，再依照優先級排序。

**編號組成：**

- `[spec-xxxxx]`：來源 Spec 的編號（5 碼）
- `[plan-yyyyy]`：計畫自己的編號，包含 `plan-` 前綴（5 碼）

**優先級代碼：**

- `urgent` - 緊急
- `high` - 高
- `mid` - 中
- `low` - 低

**類別代碼：**

- `feature` - 新功能
- `bug` - 錯誤修正
- `refactor` - 重構
- `test` - 測試
- `docs` - 文件

**範例（來自 Spec Ax4m2）：**

```

2025-12-05-[spec-Ax4m2]-2-high-[plan-Az4a2]-feature-user-authentication.md
2025-12-05-[spec-Ax4m2]-1-urgent-[plan-k3B9Z]-bug-login-error-fix.md
2025-12-05-[spec-Ax4m2]-3-mid-[plan-A1b2C]-refactor-service-logic.md
2025-12-05-[spec-Ax4m2]-4-low-[plan-9zZ8M]-test-add-unit-tests.md
2025-12-05-[spec-Ax4m2]-3-mid-[plan-Mf36K]-docs-update-api-documentation.md

```

**好處：**

- 可以一眼看出 Plan 來自哪個 Spec
- 方便批次處理或搜尋特定 Spec 的所有 Plans
- 保持實作計畫與規格文件的關聯性

#### 無對應 Spec 的 Plan 文件

**使用情境：**

- 臨時想到或緊急需要新增的小任務
- 沒有完整的 Spec 或規格流程
- 已經明確知道要做什麼事情，不需要完整的設計階段
- **可以在任何階段直接建立**

**命名格式：**

```

[YYYY-MM-DD]-[no-spec]-[數字]-[優先級]-[plan-xxxxx]-[類別]-簡短描述.md

```

**編號組成：**

- `[YYYY-MM-DD]`：Plan 建立的日期，**必須放在最前面**
- `[no-spec]`：固定標記，表示此 Plan 沒有對應的 Specs
- `[數字]`：優先級數字（0-5）
- `[優先級]`：優先級代碼
- `[plan-xxxxx]`：Plan 自己的 5 碼隨機編號
- `[類別]`：類別代碼
- `簡短描述`：使用小寫英文，單字間用 `-` 連接

**範例：**

```

2025-12-31-[no-spec]-1-urgent-[plan-Bz8k3]-bug-fix-login-timeout.md
2025-12-31-[no-spec]-2-high-[plan-Mf7n2]-feature-add-export-button.md

```

**父層資料夾規則：**

- 無論同一批次是 1 個或多個 `no-spec` plan 文件，都必須先建立共同父層資料夾，再放入所有 plan 文件。
- 建議父層資料夾命名格式：`[YYYY-MM-DD]-[no-spec]-[feature-name]/`
- 同一批次 plan 跨階段移動時，必須保留原本的父層資料夾結構。

#### Specs (規格) 文件

```

[spec-編號]-檔案名稱.md

```

**特殊規則：**

- 格式必須為 `[spec-xxxxx]`，其中 xxxxx 為 5 碼隨機編號
- **同一個 Spec 資料夾內的所有文件必須使用相同的編號**

**範例（同一 Spec 資料夾）：**

```

1-Specs/2025-12-04-[spec-Rt7w3]-migrate-to-react-native-elements/
├── [spec-Rt7w3]-IDEA_DESCRIPTION.md
├── [spec-Rt7w3]-API_DESIGN.md
├── [spec-Rt7w3]-DATABASE_SCHEMA.md
├── [spec-Rt7w3]-ARCHITECTURE.md
└── [spec-Rt7w3]-CLEANUP_AND_INTEGRATION.md

```

### 編號產生流程

**AI 工具建立 Plan 檔案時：**

1. **讀取來源 Spec 的編號**
   - 從對應的 Spec 資料夾中讀取
   - 提取 `[spec-xxxxx]` 編號部分

2. **產生 Plan 自己的 5 碼隨機編號**
   - 每一位從 `A-Za-z0-9` 中隨機選擇

3. **建立檔案**
   - 使用格式：`[YYYY-MM-DD]-[spec-xxxxx]-[數字]-[優先級]-[plan-yyyyy]-原檔名.md`

**AI 工具建立 Specs 規格文件時：**

1. **第一個文件：產生 Spec 編號**
   - 產生 5 碼隨機編號，格式：`[spec-編號]`

2. **後續文件：重複使用相同編號**
   - 檢查同一 Spec 資料夾內是否已有文件，使用相同的 `[spec-編號]` 前綴

## 資料夾組織規則

### 📁 Spec 資料夾結構

⭐ **重要原則：為避免不同 Spec 的任務混在一起，所有階段的 Plans 都必須按照所屬 Spec 組織**

### 基本規則

1. **specs 階段的資料夾命名格式：** `[YYYY-MM-DD]-[spec-xxxxx]-[feature-name]`

   - 日期：專案或功能開始的日期，**必須放在最前面**
   - 編號：Spec 的 5 碼編號
   - 名稱：使用小寫英文，單字間用 `-` 連接

2. **其他階段（2-Plans...7-Done）的資料夾命名格式：** `[YYYY-MM-DD]-[spec-xxxxx]-[feature-name]`

   - **必須與 1-Specs 資料夾名稱完全一致**
   - 使用完整的日期、Spec 編號和 feature-name
   - 不可省略 feature-name

3. **Plan 文件必須放在對應的 Spec 資料夾中**
   - 不可將 Plan 直接放在階段資料夾的根目錄
   - 若為 `no-spec` plan 文件，也必須先建立共同父層資料夾

### 資料夾結構範例

```

專案根目錄/
├── 1-Specs/
│ ├── 2025-12-01-[spec-Rt7w3]-user-authentication/
│ │ ├── [spec-Rt7w3]-IDEA_DESCRIPTION.md
│ │ └── [spec-Rt7w3]-API_DESIGN.md
│
├── 2-Plans/
│ ├── 2025-12-01-[spec-Rt7w3]-user-authentication/
│ │ ├── 2025-12-01-[spec-Rt7w3]-2-high-[plan-Az4a2]-feature-jwt-implementation.md
│ │ └── 2025-12-01-[spec-Rt7w3]-3-mid-[plan-k3B9Z]-test-auth-flow.md
│
├── 3-Progressing/
│ ├── 2025-12-01-[spec-Rt7w3]-user-authentication/
│ │ └── 2025-12-01-[spec-Rt7w3]-2-high-[plan-Az4a2]-feature-jwt-implementation.md
│
└── 7-Done/
└── 2025-12-01-[spec-Rt7w3]-user-authentication/
└── ...

