## 🎯 資料夾用途

此資料夾存放**測試失敗需要修正**的任務。當任務在 `4-Testing/` 階段測試失敗時，會移到這裡進行程式碼修正和重新測試。

### 📁 資料夾組織規則

⭐ **重要:** 為避免不同 plan 的任務混在一起，任務必須按照所屬 plan 組織：

1. **任務必須放在日期-plan 編號-feature 名稱資料夾中**
   - 例如: `5-Re-testing/2025-12-01-[plan-Xm1q2]-payment-integration/2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-stripe-integration.md`
2. **從 4-Testing 移動任務時，保持相同的日期和 plan 資料夾結構**
   - `4-Testing/2025-12-01-[plan-Xm1q2]-payment-integration/xxx.md` → `5-Re-testing/2025-12-01-[plan-Xm1q2]-payment-integration/xxx.md`
3. **⚠️ 資料夾命名格式:** `[YYYY-MM-DD]-[plan-xxxxx]-[feature-name]`
   - ❌ 錯誤: `5-Re-testing/2025-12-01-[plan-Xm1q2]/`（缺少 feature-name）
   - ✅ 正確: `5-Re-testing/2025-12-01-[plan-Xm1q2]-payment-integration/`

**範例結構:**

```
5-Re-testing/
├── 2025-12-01-[plan-Xm1q2]-payment-integration/
│   └── 2025-12-01-[plan-Xm1q2]-2-high-[task-Az4a2]-feature-stripe-integration.md
├── 2025-12-05-[plan-Rt7w3]-user-authentication/
│   └── 2025-12-05-[plan-Rt7w3]-1-urgent-[task-Bc8K7]-feature-jwt-implementation.md
├── 2025-12-31-[no-spec]-bugfix-login/
│   └── 2025-12-31-[no-spec]-1-urgent-[task-Bz8k3]-bug-fix-login-timeout.md
└── .re-testing-task-template.md
```

### 📁 無對應 Spec 的任務

對於使用 `[no-spec]` 命名格式的任務：

- **無論是單一任務文件或多個任務文件，都必須放在共同父層資料夾中**
- **可以直接在 5-Re-testing/ 建立**
- **跨階段移動時必須保留父層資料夾結構**

**詳細規範：** 請參考 [COMMON_CONVENTIONS.md](../COMMON_CONVENTIONS.md#無對應-spec-的-plan-文件)

## 📋 AI 工具應該做什麼

當任務從 `4-Testing/` 因測試失敗移到 `5-Re-testing/` 時：

1. **將任務檔案從 `kanban/4-Testing/` 移動（mv）到 `kanban/5-Re-testing/`**
2. **更新任務檔案的內容**（見下方修改規範）
3. **🚪 Stage Entry Gate（強制）**
   - 確認來源階段（`4-Testing/`）中該批次文件已不存在（已被 `mv` 移走，非 `cp`）。
   - 若來源資料夾已無同批次文件，刪除空資料夾。
   - 詳見 [COMMON_CONVENTIONS.md - Stage Entry Gate](../COMMON_CONVENTIONS.md#-stage-entry-gate階段進入閘門強制)
4. **分析測試失敗原因**
5. **修正程式碼**
6. **重新執行測試**
7. **✅ Stage Exit Checkpoint（強制）**
   - 離開 `5-Re-testing` 前必須確認：所有修正與測試記錄已填寫、Checkbox 皆為 `[✓]`、若有 PLAN_OVERVIEW 已更新。
8. **測試通過後，將任務移到 `kanban/7-Done/`**
   - 移入 `7-Done` 後也要執行該階段的 Entry Gate 來源清理。
9. **停下動作，等待確認**

## 🔧 修正任務的流程

1. **閱讀測試失敗記錄**，理解失敗原因

2. **分析根本原因**
   - 是程式碼邏輯錯誤？
   - 是測試案例寫錯？
   - 是環境或配置問題？

3. **制定修正方案**
   - 確定需要修改哪些檔案
   - 規劃修正步驟

4. **執行修正**
   - 修改程式碼或測試檔案
   - 記錄所有修改內容

5. **重新執行測試**

   ```bash
   npm test
   # 或其他測試指令
   ```

6. **記錄修正結果**
   - 在「修正記錄」中記錄此次修正的詳細資訊
   - 更新測試統計數據

7. **判斷下一步**
   - **如果測試通過:** 將任務移到 `kanban/7-Done/`
   - **如果仍失敗:** 繼續留在 `5-Re-testing/`，開始下一輪修正

---

## ⚠️ 注意事項

### 修正次數追蹤

- **第 1-2 次失敗:** 正常情況，繼續修正
- **第 3-4 次失敗:** 需要重新審視問題，可能需要重新設計
- **超過 5 次失敗:** 建議將任務移到 `6-On-hold/`，重新討論需求或設計

### 測試失敗的常見原因

1. **程式碼邏輯錯誤** - 功能實現有 bug
2. **測試案例錯誤** - 測試本身寫錯了
3. **邊界條件未處理** - 沒有考慮特殊情況
4. **非同步處理問題** - 非同步操作沒有正確等待
5. **環境依賴問題** - 測試依賴特定環境配置

### 何時該放棄修正？

如果遇到以下情況，建議將任務移到 `6-On-hold/`：

- 連續修正 5 次仍失敗
- 發現需求描述本身有問題
- 發現需要重構其他相依模組
- 發現技術方案不可行

## 📈 Re-Testing 統計指標

定期檢視 `5-Re-testing/` 資料夾，可以：

1. **追蹤程式碼品質** - 重新測試次數越少代表品質越好
2. **識別常見錯誤模式** - 分析哪些類型的錯誤最常見
3. **評估 AI 工具效能** - 哪個 AI 工具的程式碼較少進入 re-testing
4. **改進開發流程** - 根據失敗原因優化開發流程

---

## ✅ 完成後的動作

測試通過並準備移到 `7-Done/` 時：

1. 確認任務檔案已更新所有重新測試相關欄位
2. 確認所有修正記錄都已完整填寫
3. 確認最終測試統計已記錄
4. **✅ 確認 Stage Exit Checkpoint 條件已滿足** (詳見 [COMMON_CONVENTIONS.md](../COMMON_CONVENTIONS.md#-stage-exit-checkpoint階段離開檢查點強制))
5. 將任務從 `5-Re-testing/` **移動（mv）**到 `7-Done/`
6. 停下動作，等待確認
