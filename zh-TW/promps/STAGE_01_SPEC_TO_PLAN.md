# 階段 1: 從規格建立實作計畫 (1-Specs → 2-Plans)

> **📌 重要提醒:**
>
> **開始使用本提示詞前,請務必先閱讀 [KANBAN_INSTRUCTION.md](../../KANBAN_INSTRUCTION.md)**
>
> 本階段必須遵循的核心原則:
>
> - ⭐ 計畫並行執行原則 (最大化獨立計畫)
> - 📋 計畫命名規範 ([spec-xxxxx] → [plan-xxxxx])
> - 📋 停下動作等待確認原則

---

## 🎯 目標

請閱讀 `[專案名稱]/1-Specs/[專案名稱]/` 資料夾中的所有規格文件，並根據文件內容建立實作計畫卡片。

**注意：本提示詞僅用於計畫拆分與建立階段，不包含程式碼實作。**

### 你的任務流程：

1. **閱讀所有規格文件 (Specs)**
   - 仔細閱讀 `1-Specs` 資料夾中的所有 .md 文件
   - 理解專案的功能需求與系統設計
   - 提取文件中的 `[spec-xxxxx]` ID

2. **拆分實作計畫 (Plans)**
   - 將規格需求拆分為獨立的、可執行的實作計畫
   - 每個計畫應該是一個明確的工作單元（建議開發時間：4-8 小時）
   - 必須明確標註每個計畫對應的 `[spec-xxxxx]` 來源

   **⭐ 重點：最大化計畫並行執行能力**

   為了讓多個 AI 工具能**同時**處理不同的計畫，拆分時必須遵循以下原則：

   **a) 優先拆分成獨立計畫（可並行執行）**
   - 盡可能將需求拆分成**互不依賴**的計畫
   - 相同層級的功能（如多個 API 端點、多個 UI 組件）應拆分為獨立計畫

   **b) 獨立性評估標準**
   - ✅ 不需要讀取或修改其他待辦計畫會建立的檔案
   - ✅ 不需要呼叫其他待辦計畫會實作的函數或 API
   - ✅ 不需要等待其他計畫的資料結構或介面定義

   **c) 最小化依賴關係**
   - 優先完成「基礎設施計畫」（如資料模型），解除後續依賴

3. **為每個計畫評估以下資訊**
   - **類別**: feature / bug / refactor / test / docs
   - **優先級**:
     - **Urgent (1)**: 阻塞其他計畫的基礎設施
     - **High (2)**: 核心功能
     - **Mid (3)**: 一般功能
     - **Low (4)**: 文件、非關鍵優化

4. **建立計畫卡片檔案**
   - **使用檔案命名規範:**
     `[YYYY-MM-DD]-[spec-xxxxx]-[數字]-[優先級]-[plan-yyyyy]-[類別]-簡短描述.md`

   - **欄位說明:**
     - `[spec-xxxxx]`: 來源規格 ID
     - `[數字]`: 1-5 (對應優先級)
     - `[plan-yyyyy]`: 5 碼隨機編號 (本計畫 ID)
     - `[類別]`: feature, bug, refactor, test, docs

   - **範例:**
     `2025-02-10-[spec-Ax4m2]-2-high-[plan-Az4a2]-feature-auth-api.md`

5. **使用計畫模板**
   - 參考 `templates/2-Plans/.plan-template.md` (或使用通用模板)
   - **必須填寫的欄位：**
     - 計畫名稱
     - 參考文件 (列出 [spec-xxxxx] 路徑)
     - 計畫描述 (詳細說明要做什麼)
     - **依賴關係** (重要！用於判斷並行)
     - **是否可並行執行**: [是 / 否]

6. **先確認父層資料夾，再放入所有計畫卡片**
   - 無論本次建立的是 1 個或多個 plan 文件，都必須先在 `[專案名稱]/2-Plans/` 建立共同父層資料夾
   - 若此批次來自 Spec，父層資料夾必須使用與 Spec 對應的同名資料夾

### 完成後：

當你建立好所有計畫卡片後：

1. **建立計畫拆分摘要文件**
   - 檔案名稱：`PLAN_BREAKDOWN_YYYY_MM_DD.md`
   - 檔案位置：放在 `1-Specs` 資料夾中
   - 內容包含：任務清單、依賴圖、並行建議

2. **停下動作，等待確認**，不要自動開始處理計畫
